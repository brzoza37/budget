# Internationalisation (i18n) — Design Spec

**Date:** 2026-04-30
**Scope:** Add English and Polish translations to the budget PWA — frontend UI strings, backend error messages, and user locale preference stored in the user profile.

---

## Goal

Users can switch between English and Polish. The selected language persists in their profile (survives logout, works across devices). Backend error messages are also translated so users always see errors in their chosen language.

## Supported Locales

- `en` — English (default fallback)
- `pl` — Polish

## Architecture

**Approach:** react-i18next (frontend) + Symfony Translation component (backend). Each system handles its own strings. The frontend sends `Accept-Language` on every API request so Symfony returns errors in the correct language.

---

## Backend

### User entity

Add `locale` field to `App\Entity\User`:
- Type: `string`, length 5
- Default: `'en'`
- Included in auth responses (`/api/auth/register`, `/api/auth/login`, `/api/auth/me`)
- Updatable via a new `PATCH /api/auth/me` endpoint (also used by the themes feature for `currency` and theme preference)

### Locale detection on registration

Registration payload accepts an optional `locale` field. If omitted, the backend defaults to `'en'`. The frontend always sends it (detected from browser).

### Symfony Translator

- Install (already included in Symfony skeleton): `symfony/translation`
- Translation files: `translations/messages.en.yaml` and `translations/messages.pl.yaml`
- `framework.yaml`: set `default_locale: en` and `fallbacks: [en]`
- All controller error strings replaced with `$this->translator->trans('error.key')`
- Symfony reads `Accept-Language` header automatically via `LocaleListener` — no manual locale-setting code needed in controllers

### Error key naming convention

```

error.auth.invalid_email
error.auth.password_too_weak
error.auth.email_taken
error.auth.invalid_credentials
error.plan.confirm_failed
error.general.not_found
error.general.unauthorized
```

## Frontend

### Library

`i18next` + `react-i18next` — React standard, supports runtime language switching without page reload.

### File structure

```
src/i18n/
  index.ts          — i18next initialisation
  en.json           — English strings
  pl.json           — Polish strings
```

### Initialisation

`src/i18n/index.ts` initialises i18next synchronously before the React app mounts. Default language: `'en'`. `AuthContext` calls `i18next.changeLanguage(user.locale)` whenever a user loads or logs in.

### Locale detection on registration

`Register.tsx` detects `navigator.language.startsWith('pl') ? 'pl' : 'en'` and sends it as `locale` in the registration payload.

### Axios interceptor

`apiClient.ts` request interceptor adds `Accept-Language: <i18next.language>` to every request. This ensures Symfony returns errors in the user's current language.

### Translation usage

All UI strings use the `useTranslation` hook:
```typescript
const { t } = useTranslation();
// ...
<Typography>{t('dashboard.totalBalance')}</Typography>
```

### Language switcher in Settings

Settings page gets a two-option toggle (EN / PL). On change:
1. `i18next.changeLanguage(locale)` — UI switches immediately, no reload
2. PATCH user profile to persist the new locale
3. Axios interceptor automatically picks up the new language for subsequent requests

## Error Handling

| Scenario | Behaviour |
|---|---|
| PATCH locale fails | Show error toast, revert toggle to previous value |
| i18next fails to load | Falls back to key name (i18next default) |
| Missing translation key | Falls back to English (`fallbacks: [en]`) |

## Out of Scope

- More than 2 languages
- Right-to-left (RTL) layout support
- Translating date/number formats (Intl API already uses browser locale)
- Backend entity content (account names, category names — user-entered, not translated)
- Email notifications
